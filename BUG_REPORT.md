## 🔐 Безопасность

### BUG-001 🔴 Загрузка .env не автоматическая
**Описание:** `loadConfig()` читает `process.env`, но файл `.env` не загружается автоматически. При запуске через cron без експорта переменных — ключи не подтянутся.
**Сценарий:** cron не наследует env шелла → `apiKey = ''` → капча не решается тихо, нет ошибки до момента вызова solver.
**Решение:** Добавить загрузку dotenv в `loadConfig()`. Либо в cron-команде: `source /path/.env && npx tsx ...`
**Файл:** `packages/core/config.ts`

---

### BUG-002 🔴 Отсутствует валидация ключей при старте
**Описание:** Если `RUCAPTCHA_API_KEY` пуст и все провайдеры недоступны, оркестратор всё равно запустится. Ошибка проявится только при обработке magistrate-дел.
**Решение:** В `loadConfig()` добавить проверку: если есть enabled magistrate-суды и оба ключа пусты — логировать `warn` и пропускать magistrate-парсинг.
**Файл:** `packages/core/config.ts`, `packages/scheduler/orchestrator.ts`

---

### BUG-003 🔴 config.json доступен по сети через viewer
**Описание:** `GET /api/config` возвращает полный объект AppConfig. В AppConfig попадают `apiKey` и `fallbackApiKey`, подтянутые из `.env`. Если viewer доступен даже локально — ключи утекают в ответ.
**Решение:** `GET /api/config` должен возвращать конфиг без полей `apiKey`/`fallbackApiKey`. Ключи никогда не покидают сервер.
**Файл:** `packages/viewer/server.ts`

## ⚠️ Надёжность и стабильность

### BUG-004 🔴 Отсутствует timeout на парсинг
**Описание:** `fetchHtml` имеет timeout, но `adapter.parse()` — нет. Если Cheerio виснет на неправильном HTML (например если вернулась страница капчи) — оркестратор зависнет.
**Решение:** Обернуть `adapter.parse()` в `Promise.race([parse(), timeout(5000)])`.
**Файл:** `packages/scheduler/orchestrator.ts`

### BUG-005 🔴 run-log.json не аккумулирует историю
**Описание:** Каждый запуск перезаписывает `run-log.json` целиком. История запусков не сохраняется. Проблема при отладке и анализе сбоев.
**Решение:** Записывать в `run-log-YYYY-MM-DD.json` (один файл в день). Либо добавлять запись в массив если файл уже есть.
**Файл:** `packages/scheduler/orchestrator.ts`

### BUG-006 🔴 Отсутствует защита от повторного парсинга
**Описание:** Каждый запуск перезаписывает `cases-{court}-{date}.json` целиком. При парсинге в тот же день дважды — второй запуск сотрёт результаты первого.
**Решение:** Добавить timestamp в имя файла (`cases-{court}-{date}-{hhmm}.json`) или мержить дела в существующий файл.
**Файл:** `packages/exporter/json.ts`

### BUG-007 🔴 Нет lock-файла от параллельного запуска
**Описание:** Если cron запустит оркестратор пока предыдущий ещё работает (долгий запуск) — два процесса будут писать в один файл одновременно.
**Решение:** Создавать `logs/orchestrator.lock` при старте, удалять при завершении. При старте если lock есть — выход с `process.exit(0)`.
**Файл:** `packages/scheduler/orchestrator.ts`

## 🧟‍♂️ DistrictAdapter — HTML-парсинг

### BUG-008 🔴 Непроверенные CSS-селекторы
**Описание:** `DistrictAdapter` использует `#cont1`, `div.casenumber`, `div.title`, `#tablcont`, `div.publishInfo` — всё это предположения без анализа реального HTML. Если структура отличается — uid не найдется, бросается Error.
**Решение:** Проанализировать HTML реального дела перед запуском. Ссылки для анализа: sverdlov--perm.sudrf.ru из urls2.txt.
**Приоритет:** Высокий. Блокирует запуск smoke-testа.
**Файл:** `packages/adapters/district.ts`

### BUG-009 🔴 UID ищется по ссылке `judicial_uid` — возможно несуществующей
**Описание:** `$('#cont1 a[href*="judicial_uid"]').text()` — если такой ссылки нет в HTML, uid = `''` → бросается Error. UID может быть `case_uid` из URL.
**Решение:** Fallback: если `a[judicial_uid]` не найден — использовать `case_uid` из URL.
**Файл:** `packages/adapters/district.ts`, функция `parse()`

### BUG-010 🔴 Нет обработки страницы капчи или 503
**Описание:** `fetchHtml` бросает `Error: HTTP 503` — это попадает в retry и затем в FAIL. Но если вернулась страница капчи (HTTP 200 + форма вместо карточки) — `uid` не найдется, FAIL тоже, но сообщение будет вводящим в заблуждение.
**Решение:** В `parse()` проверять наличие формы капчи из передавать отдельный тип ошибки `CaptchaRequiredError` (vs `ParseError`).
**Файл:** `packages/adapters/district.ts`, `packages/adapters/magistrate.ts`

## 🔄 Зависимости и совместимость

### BUG-011 🔴 node-fetch v3 требует ESM
**Описание:** `node-fetch@3` — ESM-only пакет. Проект `"type": "module"` — OK. Но на Windows с некоторыми версиями tsx возможны проблемы с `import` наследования.
**Решение:** В smoke-test проверить `import fetch from 'node-fetch'`. Альтернатива — заменить на `undici` (встроен в Node 22) или глобальный `fetch` (Node 18+).
**Приоритет:** Средний.

### BUG-012 🔴 iconv-lite: возможна utf-8 отдача на некоторых судах
**Описание:** Не все суды возвращают windows-1251. Возможна utf-8 отдача, тогда `iconv.decode(..., 'win1251')` даст кокой вместо кириллицы.
**Решение:** Читать `Content-Type` из response headers, автоопределять charset. Fallback — win1251.
**Файл:** `packages/scheduler/orchestrator.ts`, `fetchHtml()`

3. Проверить `Content-Type` заголовка на реальном ответе (charset) — закроет BUG-012
