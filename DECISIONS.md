## Технологические решения

### Язык и рантайм
Тыпоскрипт + Node.js 24 (ESM). Без компиляции — запуск через `tsx`. native `fetch` (без node-fetch).

### Хранение URL дел
`urls.txt` — плоский файл, одна строка — одно дело. Добавить дело — добавить строку. **Нет** данных дел в `config.json`.

### Определение типа суда
Автоматически по `delo_id` + домен в `detectCourtType()`. Никакой ручной настройки.

### Один адаптер — один тип суда
Изоляция. Изменения HTML на сайте — по типам, не глобально.

### Хранение данных
JSON — основной формат. XLSX — опционально, управляется флагом `exportXlsx`.

### Мержинг по uid
Повторный парсинг обновляет запись, не дублирует. `uid` — единственный ключ.

### UI — Vanilla HTML/JS
Никаких фреймворков. Сервер — Express. Ориентир: офисный Linux-сервер, не выходит за офисную сеть.

### Запуск оркестратора
Через `child_process.spawn` из viewer. Один запуск за раз — lock-файл.

## Стратегия развития

### Фаза 1 — Базовый парсинг (✅ завершена)
- district, appeal, cassation — работают
- orchestrator на `loadUrls()`
- exporter JSON с мержингом по uid
- viewer UI с таблицей дел, логами, запуском

### Фаза 2 — Справочник судов (в работе)
- `courts.json` — название, адрес, телефоны, vnkod
- `npm run enrich:courts` — автозаполнение с главных страниц судов
- vnkod автоматически из HTML карточки дела
- В UI: `shortName` вместо поддомена
- `GET /api/courts` в server.ts

### Фаза 3 — Magistrate (после изучения вёрстки)
- Puppeteer для msudrf.ru
- Captcha flow: детекция + RuCaptcha/2Captcha API
- BUG-010: детекция капчи во всех адаптерах

### Фаза 4 — Инфраструктура
- systemd/pm2-сервис для Linux
- XLSX экспорт
- Уведомления (по email/Telegram) при изменении дела
