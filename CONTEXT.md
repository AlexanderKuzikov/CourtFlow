## 2026-07-01 — Старт проекта

### Предыстория

До CourtFlow существовали два прототипа:
- **SudRF-Parser** — скрипты `fetch-case.js` / `batch-parse.js`, Node.js, Cheerio, iconv-lite. Работал с районными судами `*.sudrf.ru`. Хардкодные URL, нет абстракции, нет retry.
- **Court-Viewer** — Express-сервер, отдаёт JSON как REST API, фронтенд для просмотра дел. Данные грузит из GitHub Raw.

Прототипы подтвердили что парсинг `sudrf.ru` работает. Структура HTML стабильна: `#cont1` (карточка), `#cont2` (движение), `#cont3` (стороны).

### Принятые архитектурные решения

| Решение | Выбор | Обоснование |
|---|---|---|
| Storage | JSON-файлы | Нет смысла в БД для «горячих» данных; downstream — 1С |
| Scheduler | System cron | Офисный сервер, простота, надёжность |
| Captcha primary | Session cookies (Puppeteer) | Мировые суды редко инвалидируют сессию |
| Captcha fallback | 2captcha API | Кириллица поддерживается, ~$0.5/1000; суммы мизерные |
| Captcha тип | Кириллический distorted text | Подтверждено скриншотами с `*.msudrf.ru` |
| Обновление данных | 2–3 раза в неделю | Движение дел достаточно редкое |
| Fault tolerance | Exponential backoff + run-log | Серверы судов нестабильны, частые сбои |
| Адаптеры | Раздельные per тип суда | HTML районных и мировых судов отличается и меняется независимо |
| Репо | Новый monorepo `CourtFlow` | Прототипы несут tech debt; остаются как reference |

### Целевая архитектура (v1)

```
[cron] → [orchestrator] ←→ [run-log.json]
              ↓
        [parser engine]
          ├── adapter: sudrf-district    # районные
          └── adapter: sudrf-magistrate  # мировые + captcha
              ↓
        [normalizer] → Case schema v1
              ↓
        [storage] → cases-{court_id}-{date}.json
              ↓
        [exporter] → 1С (формат TBD)
```
